# Barrier System Calls For Linux

This directory contains a small Linux kernel project that adds three custom
system calls for barrier synchronization and a user-space test program that
exercises them with multiple processes and threads.

The current workflow uses `modern_patch/apply_barrier_syscalls.sh`, which edits
a local Linux kernel source tree and generates the matching syscall numbers for
`user_modern.c`.

## Project Overview

The project adds a kernel-side barrier object. A process can create a barrier
with a required thread count, wait on that barrier from several threads, and
destroy the barrier after all waiting is complete.

The three system calls are:

| System call | Purpose |
| --- | --- |
| `barrier_init(count, timeout)` | Creates a new barrier and returns its barrier ID. |
| `barrier_wait(barrier_id)` | Blocks the calling thread until the required number of threads arrive. |
| `barrier_destroy(barrier_id)` | Removes a barrier when it is no longer busy. |

`barrier_init` returns the barrier ID directly. It does not store the ID in a
pointer argument.

The generated kernel code logs each call with `pr_info`, so the calls can be
observed through `dmesg`.

## How The Kernel Code Works

The helper script creates a new kernel directory named `barrier/` inside the
Linux source tree. It adds `barrier/barrier_sys.c` and registers it in the
kernel build.

The kernel implementation keeps barriers in a global list. Each barrier stores:

| Field | Meaning |
| --- | --- |
| `id` | Unique barrier ID returned to user space. |
| `tgid` | Process/thread-group ID that owns the barrier. |
| `count` | Number of threads required to release the barrier. |
| `arrived` | Number of threads currently waiting in this generation. |
| `generation` | Barrier cycle counter used to wake the correct waiters. |
| `timeout_ns` | Timeout value provided by user space. |
| `waitq` | Kernel wait queue used by `barrier_wait`. |
| `lock` | Per-barrier mutex. |
| `refcount` | Reference count used to avoid freeing a busy barrier. |

When the final required thread reaches `barrier_wait`, the kernel increments
the barrier generation and wakes all waiters.

## Prepare The Kernel Source

Run these commands inside the Ubuntu VM where you will build the kernel:

```bash
sudo apt update
sudo apt install -y git build-essential libncurses-dev bison flex libssl-dev libelf-dev bc dwarves fakeroot rsync patch linux-source

cd ~
mkdir -p kernel-work
cd kernel-work
tar -xf /usr/src/linux-source-*.tar.*
mv linux-source-* kernel
```

Keep the kernel source on the VM disk if possible. Building from a shared
VirtualBox folder is usually much slower.

## Apply The Barrier Syscalls

From the project directory:

```bash
cd ~/linux-project
chmod +x kernel_module/modern_patch/apply_barrier_syscalls.sh
./kernel_module/modern_patch/apply_barrier_syscalls.sh ~/kernel-work/kernel
```

The script:

1. Finds the next free x86_64 syscall numbers.
2. Adds `barrier/barrier_sys.c` to the kernel tree.
3. Adds syscall prototypes to `include/linux/syscalls.h`.
4. Adds syscall entries to `arch/x86/entry/syscalls/syscall_64.tbl`.
5. Generates `kernel_module/barrier_syscall_numbers.h`.

If `barrier_syscall_numbers.h` is missing, run the script again before
compiling `user_modern.c`.

## Build And Install The Kernel

Run these commands from the patched kernel source:

```bash
cd ~/kernel-work/kernel
cp /boot/config-$(uname -r) .config
make olddefconfig

scripts/config --disable SYSTEM_TRUSTED_KEYS
scripts/config --disable SYSTEM_REVOCATION_KEYS
make olddefconfig

make -j$(nproc)
sudo make modules_install
sudo make install
sudo update-grub
sudo reboot
```

A full kernel build can take a long time. Two hours can be normal in a VM if it
has limited CPU/RAM. To confirm that the build is still active, run `top` in
another terminal and look for `gcc`, `cc1`, `ld`, or `make` using CPU.

If you see warnings such as:

```text
drivers/net/wireguard/allowedips.c: warning: the frame size of 1040 bytes is larger than 1024 bytes
```

that warning is from the kernel's WireGuard driver, not from this barrier
syscall project. It is safe to continue if the build does not stop with an
error.

## Boot Into The New Kernel

After reboot, confirm that the running kernel is the newly built one:

```bash
uname -r
```

You can also check the boot menu if Ubuntu still boots into an older kernel.

## Build The User Test

After booting into the patched kernel:

```bash
cd ~/linux-project/kernel_module
gcc -o user_modern user_modern.c -pthread -Wall
```

Run it:

```bash
./user_modern
```

The program asks for:

| Input | Meaning |
| --- | --- |
| Timeout value in ns | Timeout passed to `barrier_init`. Use `0` for no timeout. |
| Number of synchronisations | How many times each thread calls `barrier_wait`. |

The test program creates two child processes. Each child process creates:

| Barrier | Thread count |
| --- | --- |
| `barrier1` | 20 threads |
| `barrier2` | 5 threads |

Each thread repeatedly calls `barrier_wait`, then the child destroys both
barriers.

## Watch Kernel Logs

In another terminal, run:

```bash
sudo dmesg --follow --human
```

Expected log lines include:

```text
syscall=barrier_init
syscall=barrier_wait
syscall=barrier_destroy
```

These logs confirm that the user program is reaching the custom kernel system
calls.

## Common Issues

### `barrier_syscall_numbers.h` not found

Run the modern patch helper again:

```bash
./kernel_module/modern_patch/apply_barrier_syscalls.sh ~/kernel-work/kernel
```

Then rebuild `user_modern.c`.

### `barrier_init: Function not implemented`

The user program is running on a kernel that does not contain the custom
syscalls. Reboot into the newly built kernel and check `uname -r`.

### Kernel build takes a long time

Use all available CPU cores:

```bash
make -j$(nproc)
```

In VirtualBox, assign more CPU cores and RAM if your host machine can spare
them. Keeping the source tree on the VM's virtual disk also helps.

### Old `barrier.patch` does not apply

Use the modern helper script instead. The old patch was created for an older
kernel layout and is kept only for reference.
